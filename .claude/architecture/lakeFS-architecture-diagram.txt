                            DEPENDENCY INJECTION CHAIN (Server Startup)
                              cmd/lakefs/cmd/run.go :: Run()

    1. Load Config
         ↓
    2. Create KVStore
         kvStore = kv.Open(ctx, kvParams)
         ↓
    3. Create Auth Services
         authService = auth.NewAuthService(ctx, cfg, logger, kvStore, ...)
         ↓
    4. Create Block Adapter
         blockAdapter = blockfactory.BuildBlockAdapterWithMetrics(ctx, ...)
         ↓
    5. Create Catalog (with KVStore)
         catalogConfig.KVStore = kvStore
         catalog = catalog.New(ctx, catalogConfig)
            ↓
           [Inside catalog.New()]:
            - Create RefManager (uses KVStore)
            - Create StagingManager (uses KVStore)
            - Create Graveler with managers
            - Assign to Catalog.Store
         ↓
    6. Create Controller (with Catalog)
         controller = api.NewController(cfg, catalog, authService, ...)
         ↓
    7. Start HTTP Server
         server.ListenAndServe(handler)

                            KEY DESIGN PATTERNS

┌─────────────────────────────────────────────────────────────────────────────────────────────┐
│  1. Explicit DI (No Global State)                                                           │
│     ✓ All dependencies injected via constructors                                            │
│     ✓ Easy to test with mocks                                                               │
│     ✓ Clear dependency graph                                                                │
│                                                                                             │
│  2. Interface-Based Composition                                                             │
│     ✓ Store interface allows multiple implementations (Graveler, Mock, etc.)               │
│     ✓ Decoupled layers                                                                      │
│     ✓ Each layer can be tested independently                                               │
│                                                                                             │
│  3. Functional Options Pattern                                                              │
│     ✓ ListRepositories(ctx, limit, prefix, search, after, opts...)                        │
│     ✓ Extensible without API signature changes                                             │
│     ✓ Example: WithListReposPermissionFilter(user, policies)                               │
│                                                                                             │
│  4. Partition-Based KV Storage                                                              │
│     ✓ Partition key: "kv:ref", "kv:staging", etc.                                         │
│     ✓ Enables sharding across partitions                                                    │
│     ✓ Supports CAS (Compare-And-Swap) via SetIf                                            │
│                                                                                             │
│  5. Rate Limiting Wrapper                                                                   │
│     ✓ KVStoreLimited wraps raw KVStore                                                      │
│     ✓ Prevents background operations from overwhelming system                              │
│     ✓ Critical operations use direct KVStore access                                        │
│                                                                                             │
│  6. Error Conversion Function                                                               │
│     ✓ ErrorToStatusCodeAndMsg injected into Catalog                                        │
│     ✓ Centralized error → HTTP status mapping                                              │
│     ✓ Decoupled error handling from persistence logic                                      │
└─────────────────────────────────────────────────────────────────────────────────────────────┘
